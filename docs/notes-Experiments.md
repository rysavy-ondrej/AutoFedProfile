## Datasets

### Data Sources (Parquet Collections)

All datasets used in this study are stored in columnar Parquet format and organized into three source collections according to their origin and execution context. The malware.parquet collection contains TLS connections extracted from executions in the Tria.ge malware sandbox and includes both traffic generated directly by malware samples and auxiliary background TLS connections produced by the sandbox infrastructure. This collection therefore supports both malicious traffic analysis and the study of sandbox-induced domain effects. The winapps.parquet collection comprises TLS connections generated by known Windows applications executed in a controlled environment, with application-level attribution and benign labeling, and serves as the primary in-distribution reference for model training and calibration. Finally, the soho.parquet collection captures TLS connections observed in a real operational LAN environment, reflecting naturally occurring and unlabeled traffic from multiple devices and applications; this collection is used exclusively for deployment-oriented evaluation, such as assessing alert rates and characterizing unknown variability under in-the-wild conditions.

| Name            | Description |
|-----------------|---------|
| malware.parquet | Collection of TLS connections extracted from executions within the Tria.ge malware sandbox. The dataset includes both TLS traffic generated directly by malware samples and auxiliary TLS connections originating from sandbox background processes and infrastructure. Each connection is associated with execution metadata (e.g., sample hash, family label where available, execution timestamp), enabling separation of malicious and background traffic as well as temporal filtering. |
| winapps.parquet | Collection of TLS connections generated by known Windows applications executed in a controlled Windows sandbox environment. Traffic is application-attributed and labeled as benign, providing a curated reference of normal application-layer TLS behavior. This dataset is used as the primary in-distribution source for training and calibration of anomaly-detection models. |

The dataset reflects mixed, naturally occurring traffic from multiple devices and applications without ground-truth labeling. It is used exclusively for deployment-oriented evaluation, such as alert-rate estimation and analysis of unknown variability, and is not assumed to be purely benign. |


### Datasets

The experiments are based on five TLS connection datasets that together cover benign application traffic, malicious communication, sandbox-induced artifacts, and real-world deployment conditions. The WINAPP-APP (D1) dataset contains labeled benign TLS connections generated by known Windows applications in a real environment and serves as the in-distribution reference for model training and calibration. Malicious traffic is represented by two labeled sandbox datasets: TRIAGE-MAL24 (D2), comprising malware TLS connections collected in 2024 and used for same-year out-of-distribution evaluation, and TRIAGE-MAL25 (D3), which contains malware traffic from 2025 and introduces additional temporal distribution shift. To isolate domain effects introduced by sandbox execution, the TRIAGE-BG (D4) dataset includes non-malicious background TLS traffic generated by the sandbox infrastructure. Finally, the REAL (D5) dataset captures unlabeled TLS connections from an operational network environment and is used to assess deployment realism and in-the-wild alert behavior under unknown and mixed traffic conditions.

**IID — Independent and Identically Distributed:**
Data are IID if each sample is independent of the others, and is drawn from the same underlying probability distribution as the training data:

* same traffic source
* same time period
* same environment
* same behavior-generating process

**OOD — Out-of-Distribution:**
Data are OOD if they are drawn from a different distribution than the training data.


IID tests whether the model learned the training distribution; OOD tests whether it can recognize when reality deviates from that distribution:

* IID performance → model capacity
* OOD benign drift → robustness
* Malware OOD → anomaly detection
* Sandbox OOD → methodological validity
* Real-network OOD → deployment realism

## Experiments

### E1a — Benign modeling

Autoencoders can model benign Windows TLS behavior with low FPR.
This experiment evaluates whether autoencoders can learn a stable representation of benign Windows TLS traffic under controlled conditions. Models are trained on benign Windows application data from July–August and validated on an independent subset from September to select reconstruction-error thresholds while avoiding temporal leakage. Testing is performed on benign traffic from November, which is temporally disjoint from training and validation, providing a mild but realistic distribution shift. The goal is not to detect anomalies, but to establish a best-case baseline for false positive behavior when the model is exposed only to benign traffic. Performance is measured using the false positive rate at a fixed threshold calibrated on validation data, along with reconstruction-error quantiles and per-application false positive rates. This experiment answers whether the chosen representations and models can robustly characterize normal TLS behavior without producing excessive alerts due to natural variability or minor temporal drift.

### E1b — Malware modeling

Autoencoders can model malware TLS behavior with low FPR.

Autoencoders are trained on malware connections from July–August, validated on September data for threshold calibration, and tested on malware samples from November. The objective is to determine whether malware traffic itself forms a coherent distribution that can be modeled with low false positive rates across time, independent of benign data. Rather than serving as a detection experiment, this setup probes whether malware TLS behavior exhibits sufficient regularity to be considered a stable domain or whether it undergoes significant temporal or behavioral drift. ### E2 — Malware OOD detection (2024)

Malware traffic is OOD relative to benign TLS.
This experiment evaluates whether TLS traffic generated by malware constitutes out-of-distribution (OOD) behavior relative to benign Windows TLS traffic, when models are trained exclusively on benign data. Autoencoders and baseline anomaly detectors are trained on benign Windows application traffic collected during July–August and validated on an independent benign subset from September to calibrate reconstruction-error thresholds at fixed false positive rates. The trained models are then evaluated on malware TLS connections from 2024, without any exposure to malicious samples during training or validation. The objective is to assess whether malicious communication patterns deviate sufficiently from the learned benign distribution to be detected as anomalies, rather than to perform supervised malware classification.

Performance is measured using true positive rates at fixed false positive rates (0.1%, 1%, and 5%), ensuring that detection capability is assessed at operationally meaningful alert levels, along with AUROC and precision–recall AUC to account for class imbalance. In addition, detection rates are reported per malware family to analyze variability in detectability across different malicious behaviors. This experiment establishes the core anomaly-detection capability of the proposed representations and models, demonstrating whether benign-trained autoencoders can identify malicious TLS traffic as semantically OOD under same-year conditions, prior to considering temporal drift or domain-shift effects.

### E3 — Future malware generalization

Detection degrades—but does not vanish—on future malware.
This experiment assesses the temporal generalization capability of anomaly-detection models by evaluating their performance on malware TLS traffic collected after the benign training period. Autoencoders and baseline detectors are trained exclusively on benign Windows TLS traffic from July–August and validated on a held-out benign subset from September to calibrate anomaly thresholds at fixed false positive rates. The trained models are then tested on malware samples from 2025, which represent both semantic out-of-distribution behavior (malicious communication) and temporal distribution shift relative to the benign training data.

The goal of this experiment is to quantify how detection performance changes as malware communication patterns, TLS stacks, and infrastructure evolve over time. Performance is evaluated using true positive rates at fixed false positive rates (0.1%, 1%, and 5%), along with AUROC and precision–recall AUC to account for class imbalance, and per-family detection rates to analyze heterogeneity across malware families. Rather than expecting performance parity with same-year malware detection, this experiment aims to demonstrate that while detection effectiveness degrades under temporal shift, it remains above baseline anomaly detectors, indicating that benign-trained models capture persistent structural properties of malicious TLS traffic that generalize beyond the training period.

### E4 — Sandbox domain shift audit

This experiment examines whether anomaly detection performance on malware TLS traffic is driven by malicious behavior itself or is confounded by sandbox-specific artifacts introduced by the triage execution environment. Models are trained exclusively on benign Windows TLS traffic from July–August and validated on an independent benign subset from September to calibrate anomaly thresholds at a fixed false positive rate. The trained models are then evaluated on two sandbox-derived datasets: triage background traffic, which represents non-malicious but sandbox-generated TLS communication, and triage malware traffic, which combines sandbox domain shift with malicious semantics.

The objective is to disentangle semantic out-of-distribution behavior (malware communication) from domain-induced distribution shift caused by sandbox execution. Evaluation focuses on comparing anomaly score distributions across real benign traffic, sandbox background traffic, and sandbox malware traffic, as well as on measuring separability at a fixed false positive rate calibrated on real benign data. If sandbox background traffic yields anomaly scores comparable to malware traffic, this indicates that detection is dominated by sandbox artifacts rather than malicious behavior. Conversely, consistent separation between sandbox background and malware traffic supports the presence of a genuine malicious signal beyond domain shift. This experiment provides a critical validity check, ensuring that subsequent malware detection results are not artifacts of the sandbox environment.

### E5a — In-the-wild alert rate (benign-trained model)

This experiment evaluates the behavior of anomaly-detection models when deployed on real, unannotated network traffic, in order to assess deployment realism and exposure to previously unseen variability. Models are trained on benign Windows TLS traffic from July–August and validated on an independent benign subset from September to calibrate anomaly thresholds at a fixed false positive rate. The trained models are then applied, without any adaptation, to unlabeled TLS connections collected from a real network environment. Since no ground-truth labels are available, the goal is not to measure detection accuracy, but to quantify the operational alert rate and characterize the nature of anomalous events encountered in practice.

Evaluation focuses on the number of alerts generated per day at the validation-calibrated threshold, the temporal evolution of anomaly score distributions, and the structure of the most anomalous connections through clustering based on TLS identity features. This experiment reveals whether models trained on curated benign datasets generalize to real-world conditions or whether they produce excessive alerts due to unmodeled benign variability. The results provide insight into the gap between laboratory evaluation and deployment conditions, highlighting the extent to which real networks introduce additional, unlabeled sources of distribution shift.

### E5b — In-the-wild alert rate (malware-trained model)

This experiment mirrors E5a but uses models trained exclusively on malware TLS traffic to examine how a malicious-domain representation behaves when exposed to real, unannotated network traffic. Models are trained on malware samples from July–August and validated on September malware data to calibrate thresholds, then evaluated on the same real-network dataset without labels. The purpose is to assess whether real-world traffic appears anomalous relative to the learned malware distribution, and to what extent benign or mixed traffic deviates from malware-derived patterns.

As in E5a, evaluation is based on alert rates per day, temporal stability of anomaly score distributions, and qualitative analysis of the most anomalous clusters. Comparing the outcomes of E5a and E5b helps distinguish whether high alert rates in real networks stem from insufficient benign coverage, fundamental domain mismatch, or the intrinsic heterogeneity of real traffic relative to both benign and malicious reference distributions. Together, these experiments contextualize anomaly detection results by demonstrating how models trained on different reference domains behave under realistic, unlabeled deployment conditions.

### E6 — Domain adaptation with unlabeled data (optional but strong)

This experiment investigates whether unlabeled real-network traffic can be leveraged to reduce false positives caused by domain mismatch, without degrading malware detection capability. Starting from the base models trained and calibrated in E5, a conservative domain-adaptation step is applied using unlabeled real-network data. To avoid inadvertently learning anomalous or malicious behavior, only connections with low anomaly scores—typically the lowest 70–90% according to the base model—are selected as likely normal and used for limited fine-tuning or representation adaptation. The adaptation process is intentionally constrained (e.g., small learning rate, limited epochs, partial freezing of layers) to minimize the risk of model collapse.

After adaptation, the models are re-evaluated on four datasets: benign Windows traffic (D1) to verify stability of the original benign model, malware datasets from 2024 and 2025 (D2 and D3) to assess whether detection capability is preserved, and the unlabeled real-network dataset (D5) to measure changes in operational alert rates. Evaluation focuses on the reduction in false positives on real-network traffic, changes in true positive rates on malware datasets, and the stability of false positive rates on benign Windows data. This experiment tests the hypothesis that carefully filtered unlabeled data can improve deployment realism by aligning the model with target-domain characteristics, while still retaining sensitivity to semantic out-of-distribution behavior. It provides a principled way to exploit unlabeled data and demonstrates how domain adaptation can be applied safely in anomaly detection settings.

### E7a — Per-application vs global modeling

This experiment evaluates the impact of modeling granularity on anomaly detection by comparing a single global autoencoder trained on all benign Windows TLS traffic with multiple per-application autoencoders, each trained on traffic generated by a specific Windows application. Per-application models are trained using benign data from D1, while validation and testing are performed on held-out benign Windows traffic to measure false positive behavior, and on malware traffic from D2 to assess detection capability. The objective is to determine whether specializing models to individual applications yields a more precise characterization of normal behavior, thereby reducing false positives, and to quantify the trade-off this specialization introduces with respect to generalization to malicious traffic.

Evaluation focuses on the reduction of benign false positive rates achieved by per-application models relative to a global model, as well as on any corresponding changes in malware detection rates. While finer-grained models are expected to better capture application-specific TLS patterns and reduce benign variability, they may also overfit to narrow behaviors and reduce sensitivity to malware that mimics legitimate application traffic. This experiment clarifies whether increased modeling granularity improves practical anomaly detection performance or merely shifts errors between benign and malicious classes.

### E7b — Per-family vs global modeling 

This experiment applies the same granularity analysis to malware traffic, comparing a single global malware autoencoder with per-family models trained separately on TLS traffic from individual malware families. Models are trained on D2 using per-family subsets, while validation and testing are performed on held-out malware samples to assess internal consistency, and on benign Windows traffic from D1 to evaluate cross-domain false positive behavior. The goal is to assess whether malware families exhibit sufficiently stable and distinctive TLS communication patterns to justify family-specific modeling, or whether malware behavior is too heterogeneous to be captured effectively at this level of granularity.

Performance is evaluated by measuring changes in false positive rates on benign traffic and changes in detection rates on malware traffic when using per-family models compared to a global malware model. A reduction in benign false positives would indicate that family-specific models capture narrower behavioral patterns, whereas a drop in detection performance would suggest limited generalization across malware behaviors. This experiment provides insight into the structure and variability of malware TLS communication and informs whether family-level specialization is beneficial or whether global modeling remains more robust for anomaly detection purposes.
