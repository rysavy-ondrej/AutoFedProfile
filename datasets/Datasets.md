# Datasets

The repository contains two primary TLS traffic datasets stored in Parquet format:

- **malware.parquet** – TLS connections generated by malware samples.
- **winapps.parquet** – TLS connections generated by selected Windows applications.

The schema and field definitions are described in the [Data](Data.md) specification.

Both datasets contain per-flow TLS metadata enriched with contextual information (application, system service, malware family). They are designed for exploratory analysis, profiling, and anomaly detection experiments.

|---------------|--------------------|------|------------|-----------|
| `malware`     | Malware sandbox    | 916 files / 63.4 MB | 828,171 | 2025-09-10 -- 2025-09-30 |
| `winapps`     | Windows Sandbox    | 5892 files / 63.2 MB | 29,526 | 2025-09-26 -- 2025-10-01 |

## Data Collection

### Malware Dataset

The malware dataset was generated using the **Tria.ge sandbox** environment. Each sample was executed in isolation, and its network traffic was captured and converted into structured TLS flow records.

The dataset includes:
- TLS handshake metadata
- Flow-level statistics (bytes, packets, duration)
- Malware family annotations (when available)

This dataset represents controlled execution conditions typical for sandbox analysis.

---

### WinApps Dataset

The WinApps dataset was created by executing selected Windows applications inside a **Windows sandbox environment**. The goal was to capture benign TLS behavior of:

- System services
- Background processes
- User-level applications

The dataset reflects realistic but controlled benign application activity.

# Labeling

Each TLS connection is labeled using available metadata fields. The labeling categories are:

- `system`
- `application`
- `malware`
- `unknown`

Labeling is rule-based and deterministic.

---

## malware.parquet

Label assignment is derived from:

- `meta.malware.family`  
  - Non-empty → labeled as `malware`
- `meta.system.service`  
  - Non-empty → labeled as `system`
- If both are empty → labeled as `unknown`

This enables separation of malicious connections from system background traffic within sandbox runs.

---

## winapps.parquet

Labeling is based on:

- `meta.application.process`

Rules:

- Empty or missing process → `unknown`
- If process belongs to predefined system processes:
```

{"System", "svchost.exe", "msedge.exe",
"backgroundTaskHost.exe", "Explorer.EXE",
"explorer.exe", "smartscreen.exe"}

```
→ labeled as `system`
- Otherwise → labeled as `application`

This allows distinguishing system-level background communication from application-generated traffic.

---

## Labeling Implementation

Labeling is implemented in:

`tls_profiling.exploration.connections.get_connection_label()`

The function evaluates presence of malware family, system service metadata, and process name to assign one of the four labels using priority:

1. `malware`
2. `system`
3. `application`
4. `unknown`

The logic is deterministic and reproducible.

# Dataset Characteristics

Both datasets:

- Contain structured TLS flow-level metadata
- Support categorical and numerical feature analysis
- Exhibit heavy-tailed traffic distributions
- Enable per-application and per-family behavioral profiling

They are suitable for:

- Exploratory analysis
- TLS fingerprint diversity studies
- Profiling experiments
- Density-based anomaly detection
- Autoencoder-based modeling
